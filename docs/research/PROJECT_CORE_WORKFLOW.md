# 项目核心功能流转说明

## 1. 先说结论

本项目不是一个“输入 URL 后由应用运行时代码自动完成克隆”的系统，而是一个“让 AI 代理按标准流程完成网站逆向重建”的工程模板。

当前仓库的核心能力主要分布在四层：

- `AGENTS.md`：定义代理在本项目中的行为约束、产物目录、执行策略和平台共识。
- `.claude/skills/clone-website/SKILL.md`：定义 `/clone-website` 的完整工作流，是克隆流程的主编排文档。
- `scripts/sync-skills.mjs` 与 `scripts/sync-agent-rules.sh`：把规则和技能同步到不同 AI 平台，保证多代理入口的一致性。
- `src/` 与 `public/`：提供 Next.js 16 + React 19 + shadcn/ui + Tailwind v4 的承接骨架，用来落地最终克隆结果。

换句话说，这个项目的“核心功能流转”本质上是一条编排流，不是一条传统的运行时调用链。

## 2. 当前仓库状态

从当前代码状态看，模板还处于“等待执行克隆”的基础阶段：

- `src/app/page.tsx` 仍然是占位页，只提示用户运行 `/clone-website`。
- `src/app/layout.tsx` 和 `src/app/globals.css` 仍然是默认骨架配置，还没有被目标站点的字体、颜色、动画和元数据替换。
- `docs/research/INSPECTION_GUIDE.md` 只定义了调研维度，还不是一次实际克隆后的研究产物。

这说明仓库现在交付的是“方法和脚手架”，不是某个网站的现成克隆结果。

## 3. 推荐方案与原因

本项目选择的不是“截图转代码”方案，也不是“单代理一次性重写整个页面”方案，而是下面这套推荐方案：

- 浏览器驱动调研：通过浏览器自动化工具真实访问目标站点，提取截图、DOM、样式、交互和资源。
- 规格先行：先把每个 section 的结构、样式、状态和资源写成规格文档，再开始编码。
- 并行构建：按 section 或子组件拆分任务，分派给多个 builder agent 在独立 worktree 中并行实现。
- 回合式验收：每轮合并后都重新构建，最后再做视觉比对和交互回归。

这样设计的原因很明确：

- 只看截图无法可靠提取交互、状态切换、滚动行为和真实资源路径。
- 只靠单代理一口气重建整页，容易遗漏细节，且一旦判断错交互模型，返工成本很高。
- 规格文件可以把“观察结果”和“实现结果”解耦，既方便审计，也方便多代理并行。
- worktree 并行方式可以把复杂页面拆小，降低 builder prompt 复杂度，提升像素级还原稳定性。

作为对比，可以把几种方案理解为：

1. 截图直出代码：速度快，但精度最低，几乎无法覆盖真实交互和资源分层。
2. 单代理整页实现：流程简单，但 prompt 过长，容易近似实现，不利于排错。
3. 本项目方案：前期调研成本更高，但可审计、可并行、可回溯，最适合高保真克隆。

## 4. 核心工作流总览

项目的主工作流可以概括成下面 8 个阶段：

1. 命令入口与平台分发
2. 预检与环境确认
3. 站点侦察与全局调研
4. 基础层改造
5. Section 级规格提取
6. 并行构建与合并
7. 页面总装
8. 视觉验收与回补

下面按阶段展开。

## 5. 每一步具体怎么做

### 5.1 命令入口与平台分发

入口命令是：

```text
/clone-website <target-url1> [<target-url2> ...]
```

这一层的关键不是应用代码，而是技能分发机制：

- Claude 侧的源文件是 `.claude/skills/clone-website/SKILL.md`。
- Codex、Copilot、Cursor、Gemini、Continue 等平台使用 `scripts/sync-skills.mjs` 生成各自格式的命令文件。
- `AGENTS.md` 是项目规则单一真源，`scripts/sync-agent-rules.sh` 会把它展开并同步到其他代理平台。

这一步的目标，是确保不管用户从哪个 AI 平台触发命令，最终拿到的都是同一套克隆流程定义。

### 5.2 预检与环境确认

真正开始克隆前，技能要求先做预检：

1. 检查浏览器自动化能力是否可用，优先 Chrome MCP，没有就退到 Playwright 一类工具。
2. 校验 URL 是否可访问，避免后续调研建立在错误输入上。
3. 先运行一次 `npm run build`，确认 Next.js 脚手架本身是健康的。
4. 准备输出目录，例如 `docs/research/`、`docs/design-references/`、`public/`。

这么做的原因是：

- 没有浏览器自动化，就无法完成真实页面侦察。
- 基础工程如果一开始就是坏的，后续所有调研和 builder 输出都会失去落点。
- 目录先建好，后续调研、资源下载和规格沉淀才有稳定出口。

### 5.3 站点侦察与全局调研

这是整个流程最关键的调研阶段，主要做四类事情。

第一类是截图基线：

- 采集桌面端全页截图。
- 采集移动端全页截图。
- 必要时补充关键交互态截图，例如 hover、dropdown、modal、tab 切换态。

第二类是全局设计令牌提取：

- 字体：检查页面真实使用的字体族、字重、字形和加载方式。
- 颜色：提取背景、正文、强调色、边框、阴影等实际计算值。
- 元数据与图标：下载 favicon、apple-touch-icon、OG 图。
- 全局行为：识别滚动容器、滚动平滑库、全局动画、毛玻璃、叠层模式等。

第三类是行为扫描：

- Scroll sweep：慢速滚动整页，记录 header、section、sticky 区块、滚动驱动切换等行为。
- Click sweep：逐个点击按钮、tab、pill、card、link，记录每个状态对应的内容变化。
- Hover sweep：逐个悬停按钮、卡片、导航项，记录颜色、阴影、位移、透明度等变化。
- Responsive sweep：至少检查 `1440px`、`768px`、`390px` 三档宽度，记录布局何时切换。

第四类是页面拓扑梳理：

- 给页面从上到下所有 section 命名。
- 区分哪些是普通流式内容，哪些是 fixed 或 sticky 叠层。
- 标注每个 section 的交互模型，是静态、点击驱动、滚动驱动，还是时间驱动。

这一阶段的主要沉淀物应该是：

- `docs/research/BEHAVIORS.md`
- `docs/research/PAGE_TOPOLOGY.md`
- `docs/design-references/*.png`

本质上，这一步是在回答“页面长什么样”和“页面为什么会那样动”。

### 5.4 基础层改造

全局侦察完成后，不直接开始写 section，而是先改造基础层。这个顺序是刻意设计的。

主要改造内容有：

- 在 `src/app/layout.tsx` 中替换成目标站点字体和 metadata。
- 在 `src/app/globals.css` 中写入目标站点的色板、圆角、阴影、间距、动画和全局滚动行为。
- 在 `src/types/` 中定义后续页面内容和 section 数据结构。
- 在 `src/components/icons.tsx` 中沉淀页面里真实出现的 SVG 图标。
- 生成并执行 `scripts/download-assets.mjs`，把图片、视频、favicon 等资源下载到 `public/`。

这一步必须先做，原因是后面的 section builder 都依赖统一的基础层：

- 字体不对，后面 section 再怎么抠样式也会失真。
- 颜色和动画没统一，builder 会反复重复实现相同基础规则。
- 资源没先下载到本地，组件只能先写远程 URL，后续会导致替换和验收混乱。

从工程角度看，这一层把“克隆页面”变成“在一个已经具备目标站点设计语言的骨架里装配组件”。

### 5.5 Section 级规格提取

这一步是本项目区别于普通“读页面然后写代码”的核心做法。

对于页面中的每一个 section，需要按下面顺序处理：

1. 单独滚动到目标 section，截取局部截图。
2. 用浏览器里的 DOM 和 `getComputedStyle()` 批量提取容器、文本、图片、卡片、按钮等元素的精确计算样式。
3. 对有状态的元素做多态采集，例如滚动前后、hover 前后、tab 切换前后。
4. 提取真实文本、alt、aria 和状态切换后的内容。
5. 标记这个 section 依赖了哪些本地资源和图标。
6. 评估复杂度，判断是一个 agent 做完整 section，还是拆成多个子组件分别做。

每个 section 在派工前都必须先落成规格文件：

```text
docs/research/components/<component-name>.spec.md
```

规格文件至少要覆盖这些内容：

- 目标组件路径
- 对应截图路径
- DOM 结构
- 每个关键元素的计算样式
- 所有状态与触发条件
- 每个状态下的真实文案和资源
- 响应式变化
- 具体实现建议

这一步的设计价值在于：

- builder 拿到的是完整、可执行、可核对的规格，不需要靠记忆补全细节。
- 如果最后视觉不一致，可以先检查规格是否提取错，再检查实现是否写错，问题定位更清晰。

### 5.6 并行构建与合并

规格文件写完后，才进入真正的实现阶段。

并行策略如下：

- 简单 section：一个 builder agent 负责整个 section。
- 复杂 section：拆成多个子组件 builder，再由一个 wrapper builder 负责组装。
- 每个 builder 在自己的 git worktree 中工作，减少互相覆盖。

每个 builder 收到的不是一句“去实现这个 section”，而是一组完整输入：

- 规格文件全文
- 对应截图路径
- 目标文件路径
- 可复用的共享能力，例如 `icons.tsx`、`cn()`、shadcn 原语
- 必须通过 `npx tsc --noEmit` 的约束

主代理在这一阶段要持续做两件事：

- 一边继续提取后续 section 并派工。
- 一边把已完成的 worktree 分支合并回主分支，并在每次合并后运行 `npm run build`。

这形成了一个回环：

```text
extract -> spec -> dispatch -> merge -> build verify
```

它的意义是把“调研”和“实现”串成流水线，而不是先整站调研完再一次性写代码。

### 5.7 页面总装

等所有 section 都完成后，再回到 `src/app/page.tsx` 做页面总装。

这一阶段主要做：

- 引入全部 section 组件。
- 按 `PAGE_TOPOLOGY.md` 定义真实的页面顺序和层级关系。
- 连接 page-level 行为，例如 scroll snap、IntersectionObserver、sticky 布局、滚动驱动动画、平滑滚动。
- 将提取出的真实内容或结构化数据接入到组件 props。

这一步其实是在把前面分段完成的构件，重新拼成一张完整页面。

### 5.8 视觉验收与回补

最后一阶段不是“看起来差不多就结束”，而是做差异回补。

验收方法包括：

1. 原站与克隆页按同一视口截图。
2. 桌面端逐段比对。
3. 移动端再逐段比对。
4. 重新滚动、点击、hover，核对交互是否和原站一致。

发现差异后，按下面顺序排查：

1. 规格提取是否错了。
2. 规格没错但 builder 实现错了。
3. 资源路径、字体加载、断点切换或 page-level 行为是否遗漏。

只有做完这一轮回补，项目才算完成一次真正的克隆。

## 6. 关键产物与职责分工

这条工作流会持续产出几类关键文件：

- `docs/design-references/`：截图基线，负责“看起来是否一致”。
- `docs/research/BEHAVIORS.md`：行为总表，负责“页面为什么这样动”。
- `docs/research/PAGE_TOPOLOGY.md`：页面组装蓝图，负责“页面如何拼起来”。
- `docs/research/components/*.spec.md`：组件规格单，负责“builder 应该实现什么”。
- `public/images/`、`public/videos/`、`public/seo/`：本地资源缓存，负责“资源真实可复现”。
- `src/components/` 与 `src/app/page.tsx`：最终交付代码，负责“把调研结果落地成可构建页面”。

从职责角度看，可以把它理解为三类角色协作：

- 主代理：负责调研、写规格、派工、合并、验收。
- builder agents：负责按规格实现具体组件。
- Next.js 模板骨架：负责承接最终代码和静态资源。

## 7. 这套流转的真正优势

如果只从实现效率看，这条链路并不算最短；但如果目标是高保真、可追溯、可并行扩展，它是合理的。

它的优势主要在：

- 可审计：每个 section 都有截图和规格，误差来源可追踪。
- 可并行：复杂页面可以拆小并同时推进。
- 可回补：最终差异可以精确追溯到“提取错”还是“实现错”。
- 可迁移：技能和规则通过同步脚本分发到多个 AI 平台，不绑死单一代理。

## 8. 总结

本项目的核心功能流转可以总结成一句话：

先通过浏览器自动化把目标站点调研成结构化规格，再基于 Next.js 骨架用多代理并行把规格还原成页面，并用构建校验与视觉比对反复收敛。

因此，本项目最核心的不是某个 React 组件，也不是某个脚本，而是这套“调研 -> 规格 -> 派工 -> 合并 -> 验收”的工程化流水线。
